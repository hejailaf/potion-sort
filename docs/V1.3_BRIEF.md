4. **Latest stable APIs only.** Before writing animation or graphics code, verify the installed versions of `react-native-reanimated`, `@shopify/react-native-skia`, and Expo SDK, and use their **current** APIs (e.g., Reanimated worklets/`withTiming`/`withSpring`/layout animations; Skia declarative components). 5. **Tests ship with logic.** Every new engine function or rule gets Jest unit tests in the same phase. A phase is done only when its acceptance criteria pass and `npm test` + `npx tsc --noEmit` are clean.
6. **A phase is demonstrated, not declared.** Show me the passing tests / a simulator screenshot before moving on.
7. **Ask me before any decision.** Whenever you face a choice that isn't already settled by this brief, CLAUDE.md, or an approved plan — design trade-offs, naming that affects the player, difficulty tuning values, library/API choices, what to delete when "uncertain", visual style calls — stop and ask me with 2–3 concrete options and your recommendation. Never pick silently, and never bundle a decision inside a larger diff hoping it passes unnoticed. 8. **Use the iOS Simulator freely for testing.** You may boot the simulator, build/run the app, take screenshots, record video, and drive interactions with the available tooling (`xcrun simctl`, XcodeBuildMCP/idb if present) at any point in any phase to verify your work — no permission needed for read-only testing. ## Phase 0 — Repo audit & decluttering (do this first)

Goal: make the codebase smaller, cleaner, and cheaper for every future session.

- Inventory the repo: list every asset (images, audio, fonts), source file, config, and dependency, and classify each as **used / unused / uncertain**. Detect usage by actual imports/references, not filenames.
- Propose a deletion list in Plan Mode. Never delete anything classified "uncertain" without asking. Do the deletion as a single dedicated git commit so it's trivially revertible.
- Remove: unused assets, dead code, commented-out blocks, orphaned components, unused dependencies (`npx depcheck` as a starting signal, verified manually), stale scripts, duplicated config.
- Consolidate scattered constants (colors, timings, spacing) into a single theme/tokens module if not already done — this directly enables the Phase 4 UI overhaul.
- Add/verify `.gitignore` and an `.claudeignore`-style hygiene note in CLAUDE.md if build artifacts or large binaries are bloating context.

**Acceptance criteria:** app builds and plays identically after cleanup; deletion commit is isolated; a short `AUDIT.md` (temporary, deleted at end of v1.3) records what was removed and why.

## Phase 2 — Mechanics framework (engine)

Goal: a generic **level modifier system** so this update ships two mechanics and future updates ship more for near-zero cost.

- Extend the level schema with a `modifiers` array (typed, serializable, seeded-RNG friendly). The engine resolves modifiers; the UI merely renders their state.
- Unlock schedule is data, not code: a `progression.ts` map of `level → mechanic unlocked`. v1.3 ships: **level 20 → Mechanic A**, **level 40 → Mechanic B**. Mechanics appear in generated levels only after their unlock level, with a ramp (see below).
- **Mechanic A — Veiled Bottles (level 20):** one or more bottles start fully hidden (opaque veil, contents unknown, cannot be poured into or from). A veiled bottle reveals itself when the player completes (corks) any other bottle. Ramp: levels 20–39 go from 1 veiled bottle with early reveal opportunities → multiple veiled bottles where reveals must be sequenced deliberately.
- **Mechanic B — proposal required (level 40):** default proposal is **Mystery Potions** — segment colors below the top segment display as "?" and reveal only when they surface, forcing memory/deduction. In your Phase 2 plan, present Mystery Potions plus one alternative (e.g., chained/locked bottles requiring N pours to unseal) with a difficulty/implementation trade-off table, and let me pick before you build.
- Both mechanics: full engine implementation, deadlock detection updated to account for them, generator guarantees solvability with modifiers active, difficulty ramp expressed as tunable parameters (not magic numbers).
- Unit tests: reveal triggers, pour legality with modifiers, solvability of generated modifier levels (property-style test over many seeds), unlock schedule boundaries (levels 19/20/21, 39/40/41).

**Acceptance criteria:** all tests pass; a level JSON with each modifier round-trips serialize/deserialize; generator produces solvable modifier levels across 200 random seeds.

## Phase 3 — Mechanics UI & unlock moments

- Render Veiled Bottles (Skia veil/fog material + reveal animation) and Mechanic B's visual language, consistent with the cozy alchemy theme.
- Each mechanic gets a one-time **unlock moment**: short celebratory interstitial at level 20/40 that names the mechanic, shows a 2–3 step visual explainer, and a "got it" dismiss. Skippable, never shown twice, state persisted.
- First level featuring each mechanic includes a lightweight contextual hint (dismissable tooltip), then never again.
- Haptics + SFX hooks for reveal events, matching existing audio patterns.

**Acceptance criteria:** simulator walkthrough of levels 19→21 and 39→41 showing unlock interstitial, hint, reveal animation at 60fps, and correct persistence after app restart.

## Phase 4 — Animation & UI overhaul

Goal: refinement, not redesign. Audit first, then fix.

- Run a UI/animation audit pass in Plan Mode: list every screen and interaction with a verdict (keep / polish / rework) and specific issues (janky timing, inconsistent spacing, off-theme colors, missing feedback).
- Polish targets: pour animation feel (spring curves, anticipation/settle), bottle selection feedback, win sequence pacing, coin-fly effect, transitions between screens (use Reanimated layout/shared-element style transitions where supported by the installed version), button press states, safe-area correctness on notched devices.
- All timings/curves come from the tokens module (Phase 0) so tuning is one-file.
- Performance guardrail: profile before/after; no dropped frames during pour on an iPhone 12-class simulator; Skia usage stays limited to where plain Views can't do the job.

**Acceptance criteria:** side-by-side before/after screen recordings for the 3 biggest improvements; no regression in interaction latency; consistent spacing/typography per the tokens module.

## Phase 5 — Journey tab rework

- Redesign the Journey/level-map tab to visualize progression as a path with **milestone nodes at every mechanic unlock** (20, 40, and future slots rendered as locked/teased "???" nodes so the system scales).
- Locked mechanics show a silhouette + level requirement; unlocked ones show their icon and a tap-to-recall mini explainer (reuses the Phase 3 interstitial content).
- Current-level indicator, completed-level styling, smooth scroll-to-current on open.
- Data-driven from `progression.ts` — adding a level-60 mechanic later must require zero Journey-tab code changes.

**Acceptance criteria:** Journey tab reflects a fresh save, a level-25 save, and a level-45 save correctly (test via a debug state injector); milestone nodes come from progression data only.
